# Render spec — turning a `tutor-eval` run into the demo page

The harness emits a verdict JSON (`eval/runs/<id>.json`). The renderer turns it into a
self-contained HTML page that does two jobs:

1. **The stroll-through** — a PI or fellow instructor reads the real tutored session top to
   bottom and sees what the homework feels like, without roleplaying it themselves.
2. **The honest scorecard** — proof that this teaching is *tested*, not asserted. Shows how the
   real skill scored against its own promises, including any partials/fails. A glossy happy-path
   would be less persuasive to a skeptical instructor than visible, evidenced rigor.

## Design — sibling of `setup-guide.html`, not a clone
Same tokens so the two pages read as one workshop:
- `--ground #0F1620`, `--ground-raised #16202C`, `--ground-sunken #0A0F16`
- `--text #E8EDF2`, `--text-dim #94A3B2`, `--text-faint #5E6E7E`
- `--accent #7DE0C2` (mint = pass / the tutor's voice), `--accent-2 #F2B5A0` (coral = a fail / the student's stumbles)
- fonts: Space Grotesk display, Inter body, JetBrains Mono for labels/metadata
- self-contained (CSP: no remote anything), responsive, `prefers-reduced-motion`, visible focus.

## Required sections
1. **Header** — title, one-line "what this is," and the config as mono metadata chips
   (persona · track · lane · source). A "view the repo ↗" link and a link back to the setup
   guide / homework (this page must also be a valid standalone entry point).
2. **The honesty banner** — a short, plain statement that this is a *real simulated session*
   (a model playing the student persona against the actual skill fetched from the public repo),
   NOT a screen recording, and that the scores are an LLM-judge panel. Name the limitation; do
   not oversell. This is load-bearing for instructor trust.
3. **The transcript** — the heart. Tutor turns and student turns visually distinct (tutor =
   mint-edged, student = coral-edged or dimmer). Multiple-choice quizzes rendered as the option
   list the way they appeared. Readable as a conversation, not a data dump. Annotate 2-4 pivotal
   moments inline (e.g. "← the escape hatch fires here", "← gated: didn't advance until the why
   landed") tying the moment to the mechanism — sparingly.
4. **The scorecard** — the rubric criteria grouped by lens (Pedagogue / Guardian / Skeptic),
   each with its pass/partial/fail/not-exercised verdict and the evidence quote. Show the tally.
   Do not hide fails — show them with the evidence and (if useful) what they imply.
5. **Footer** — "where to go next" (homework, repo, setup guide), and a line that this page was
   generated by the `tutor-eval` harness and can be re-run after any skill edit.

## Rules
- Quote the transcript verbatim from the JSON; do not invent or polish student/tutor turns.
- If a criterion failed, the page says so. The whole value proposition is honesty.
- Keep the boldness in one place (the design skill's rule): the transcript is the hero; the
  scorecard is quiet and precise around it.
